Permission Modes

Cycle through modes with Shift + Tab in GUI, or \permissions in terminal.

Default

Ask Before Edits

Claude asks permission before every file change. Safe but slow. Good for: high-risk codebases, learning what it's doing. Most old-school devs prefer this.

Recommended

Edit Automatically

Auto-accepts edits to existing files. Still asks before creating new files. Good balance for most workflows. Better productivity without full danger.

Strategic

Plan Mode

Read-only. Claude can browse, research, read files — but makes zero changes. Use at the start of any complex build. Always use this first. A minute planning = 10 min saved building.

Power Users

Bypass Permissions

No prompts. Full access to create, edit, delete anything. Maximum speed. Risk: rare but possible to delete important files if Claude misinterprets a request. Enable in VS Code extension settings. Best for knowledge work.

⚠ Bypass Permissions Risk: One documented case of Claude deleting all hard drive data via a misinterpreted terminal command. Vanishingly rare, but back up important work. Enable in VS Code → Extensions → Claude Code → Settings → "Allow Dangerously Skip Permissions."

New Project Setup Ritual

Do this every single time you start a new project. Non-negotiable.

1
Create a dedicated folder — Give it a clear, descriptive name. Open it in your IDE (VS Code / Antigravity).
2
Open Claude Code & run /init — This reads your codebase (or empty folder) and auto-generates a CLAUDE.md. Don't skip this. It saves tokens every future session.
3
Review & prune the generated CLAUDE.md — Remove bloat. Add your specific conventions. Keep it under 200–500 lines. Put the most critical rules at the very top (primacy bias).
4
Set up .claude/ directory structure — Create .claude/rules/, .claude/skills/, .claude/agents/ as needed. Not all projects need all folders.
5
Choose your permission mode — For most new projects: start with Plan Mode to map the build, then switch to Edit Automatically or Bypass for execution.
6
For complex builds — run Plan Mode first — Describe what you want (voice dump is fine). Let Claude ask clarifying questions. Get a written plan/spec before touching any code.
7
Set up any needed MCP servers or skills — Paste the MCP JSON snippet into Claude. Test it works. Convert to a skill if you'll use it repeatedly (token-efficient).
8
Execute with the Task → Do → Verify loop — Give Claude a task. Let it do the task. Make it verify its result (screenshot, test, etc.). Repeat. Never skip verification.
9
Periodic context management — Watch your context % in the status bar. Run /compact proactively. If switching tasks, run /clear. Update CLAUDE.md with any new rules Claude kept getting wrong.
💡 Voice dump tip: Use a voice transcription tool to describe your project requirements. You speak ~200 wpm vs typing ~65 wpm — that's 3x faster. Then have Claude compress it into a tight spec before the build starts.

CLAUDE.md Best Practices

CLAUDE.md is injected at the top of every conversation — before your first message. It steers the entire session. Treat it like your project's operating manual for Claude.

Three Layers (merged in order)
LayerLocationScope
Global~/.claude/CLAUDE.mdApplies to ALL workspaces. Put your personal preferences, universal rules here.
Project.claude/CLAUDE.md in your folderApplies to this project only. Project-specific conventions, stack info, etc.
EnterpriseManaged system levelFor enterprise licenses only. Ignore unless relevant.
✅ DO
  • Run /init first to auto-generate a base CLAUDE.md
  • Use bullet points & short headings — high information density
  • Put critical rules & guardrails at the very TOP (primacy bias)
  • Keep it 200–500 lines max. Shorter = smarter Claude
  • Treat it as living code — prune it regularly
  • Add a rule any time Claude makes the same mistake 2–3 times
  • Use @include filename.md to reference other files
  • Split into rules files for big projects (.claude/rules/)
  • Find cutting-edge tips via Grok: "best CLAUDE.md practices last month"
❌ DON'T
  • Dump entire API docs or style guides into it
  • Write vague aspirational rules like "be smart" or "make no mistakes"
  • Exceed 500 lines — quality drops, cost rises
  • Leave it static — it will accumulate cruft over time
  • Use voice transcript dumps directly — compress first
  • Put things in CLAUDE.md that should be skills (load on demand = better)
  • Forget that everything in CLAUDE.md costs tokens every single session
💡 Primacy & Recency Bias: Claude (like humans) remembers the beginning and end of a prompt best, with the middle being weakest. Put your most important guardrails at the TOP of CLAUDE.md.

.claude Directory Structure

The .claude/ folder (hidden by default) unlocks advanced functionality. You don't need all of it — start with what you need.

Full Structure Map
.claude/
├── CLAUDE.md              # Project-level brain (injected every session)
├── CLAUDE.local.md        # Git-ignored personal overrides (secrets, local paths)
├── settings.json          # Team permissions, hooks, env variables
├── settings.local.json    # Git-ignored local settings
├── .mcp.json              # MCP server configurations
├── rules/                 # Split CLAUDE.md into focused rule files
│   ├── workflow.md        #   → How to approach tasks
│   ├── tech-defaults.md   #   → Stack preferences, libraries
│   └── design-rules.md    #   → UI/style conventions
├── skills/                # On-demand workflows (only loaded when invoked)
│   ├── scrape-leads.md    #   → Lead scraping workflow + scripts
│   └── design-website.md  #   → Website design workflow
└── agents/                # Subagent definitions
    ├── researcher.md       #   → Research subagent (uses cheaper Sonnet)
    ├── code-reviewer.md    #   → Unbiased code review (blank context)
    └── qa-tester.md        #   → Automated test generation & running
rules/

Break a big CLAUDE.md into focused files. Each rule file covers one domain. Easier to maintain, update one without touching others. Good for teams.

skills/

Only the front matter (name/description) loads in context. Full skill only loads when invoked. Huge token saving. Convert MCP workflows to skills for efficiency.

agents/

Subagents run in isolated context. Use for: research (cheap model, parallel), code review (unbiased blank slate), QA testing (automated). Pollute their own context, not yours.

Memory.md — Auto Memory

A small file injected at the start of every session — separate from CLAUDE.md. When you say "remember that…" Claude writes to this file. Persists between sessions. Think of it as Claude's personal notepad.

# Example memory.md entries
- User's brother's name is George
- User prefers serif fonts for headings
- Stripe API key pattern: sk_test_...

Skills vs MCP vs Subagents

Skills

  • You define the workflow
  • Stored as .md files in .claude/skills/
  • Only front matter loads in context (tiny token cost)
  • Uses native Claude intelligence for error handling
  • Best for: your own repeatable workflows, SOPs
  • ~60 tokens in context vs ~1600 for MCP
  • Long-term preferred over MCP

MCP Servers

  • Others build the tools (developer teams)
  • Easy 1-step install (paste JSON snippet)
  • High token cost — some eat 20%+ of context
  • Great for prototyping integrations quickly
  • Best for: testing if something is possible
  • Then convert to a skill for production use
  • Avoid bloated MCPs with 50+ tools

Subagents

  • Isolated context windows (fresh start)
  • Run in parallel — big speed gains
  • Return summary to parent (e.g. 2K tokens vs 100K)
  • Use cheaper models (Sonnet, Haiku)
  • Best 3: researcher, code reviewer, QA tester
  • Don't spawn subagents for simple queries
  • More tasks in parallel = multiply failure risk
💡 The MCP → Skill Pipeline: Install MCP → test if it can do what you want → if yes, ask Claude to rebuild it as a skill using direct API calls → remove the MCP. You'll use ~25x fewer tokens and it'll run faster.
3 Subagents Worth Always Having
AgentWhy it worksModel to use
Researcher Does web research, API lookups, doc reading in its own context — returns just a tight summary. 50x cheaper than doing it in main thread. Sonnet
Code Reviewer No prior context = no bias. Fresh eyes on the code. Catches things the parent agent missed because it was too deep in its own thinking. Sonnet
QA Tester Generates automated tests, runs them, reports failures. The equivalent of test-driven development without you having to write tests. Sonnet

Context Management

Run /context to see exactly what's eating your token budget. You're billed for everything — including system prompts you didn't write.

What Eats Your Context (in order of injection)
SourceTypical SizeYou Control?
System tools (bash, web search, task, etc.)~17,000 tokensNo
MCP tool definitions1,600–20,000+ tokensYes — choose MCPs carefully
Global CLAUDE.md (~/.claude/)VariableYes — keep short
Project CLAUDE.md + rulesVariableYes — keep short
Memory.md~88 tokensYes — usually tiny
Skills (front matter only)~60 tokens eachYes — very efficient
Your messages + tool callsGrows over timeManage actively
Active Management Strategies
  • Run /compact before context gets full — don't wait for auto-compact
  • Run /clear when switching to a totally different task
  • Use /compact [custom instructions] to tell it what to prioritize when compressing
  • Add /status-line token counter in terminal to monitor live
  • Write specific prompts — "fix this one bug in auth.js" beats "improve the codebase"
  • Compress voice transcripts before sending to Claude (use a cheap model)
Structural Token Savings
  • Use skills instead of MCP for repeated tasks (~25x token saving)
  • Move long CLAUDE.md instructions into skills (load on demand)
  • Use Sonnet/Haiku for subagents — not Opus
  • Front-load research in plan mode — avoid research during building
  • Avoid bloated MCP servers (use /context to audit)
  • Tell Claude to write concisely — "give bare bones output, ask if I need more"
  • Enable extended thinking — reasoning stays out of message chain
⚠ Context Rot Warning: Long conversations degrade Claude's output quality. There's a negative relationship between prompt length and output quality. This is why /compact and /clear aren't just about cost — they actively improve Claude's performance.

Plan Mode & Complex Builds

1. Voice dump requirements
→
2. Switch to Plan Mode
→
3. Claude asks Qs
→
4. Review the plan
→
5. Switch to Bypass
→
6. Execute
❌ Without Planning
  • Build (15 min)
  • Discover wrong approach (5 min testing)
  • Rebuild from scratch (15 min)
  • Repeat until right
  • Total: 35+ min + high token cost
✅ With Planning
  • Plan mode (5 min) — catches wrong approaches in spec
  • Refine spec (5 min)
  • Execute once, correctly (5–15 min)
  • Total: 15–25 min + lower token cost
💡 Rule of thumb: 1 minute of planning saves 10 minutes of building. Use plan mode for anything more complex than a simple design or text edit. The plan becomes a reference document Claude can use throughout the build.
Design Workflow (3 Approaches)
ApproachHowBest For
Screenshot Loop Give Claude a reference screenshot → it builds → screenshots result → compares & iterates until 99% match Replicating a design style. Gets you 80% there fast, then refines to 99%.
Voice Transcript Dump Hold FN key → speak your requirements → paste into Claude. No design reference needed. Custom builds with no design inspiration. Takes more back-and-forth.
Components Browse sites like 21st.dev → "Copy Prompt" → paste into Claude → install the component Specific UI elements (animations, buttons, sign-up forms). Mix and match.
💡 Screenshot Hack: To capture a full-page screenshot — Inspect (right-click) → set width to 1920px → Cmd+Shift+P → type "screenshot" → "Capture full size screenshot." Then resize at ilovemage.com to under ~4MB before feeding to Claude.

Hacks & Pro Tips

🗣️

Voice Transcript 3x Speed

You speak ~200 wpm but type ~65 wpm. Use Mac's FN key or any voice-to-text tool to dump requirements. Then paste into Claude and say "compress this into a high-info-density spec."

📸

Full-Page Screenshot Hack

Right-click any site → Inspect → set dimensions to 1920×1080 → Cmd+Shift+P (Ctrl+Shift+P on Windows) → "Capture full size screenshot." Gets the entire page. Resize to under 4MB before use.

🔔

Chime Hook for Multi-Tab

Set up a hook that plays a different chime sound when each Claude tab finishes. Chime 1 = tab 1 done. Chime 2 = tab 2 done. You can supervise 3–4 tabs without constantly checking.

🐦

Grok for Cutting-Edge Tips

Go to X.com → ask Grok: "compile best CLAUDE.md practices from the last month." Claude Code power users post gold daily. Grok has full Twitter access. Better than static docs.

🧹

Clean Up Workspace Bloat

Claude in bypass mode creates temp files, tries failed approaches, installs libraries. Periodically ask: "go through my workspace and delete anything that's no longer needed." Do this every few sessions.

🔁

MCP → Skill Pipeline

Use MCP to prototype a workflow (5 min setup). If it works, say: "convert this to a skill using direct API calls instead of MCP." You'll drop from ~1600 tokens to ~60 tokens per session.

⚡

Fast Mode in Terminal

Run /fast in terminal for 2.5x speed at ~3x cost. Worth it when you're in productive flow and time matters more than token cost. Not available in GUI.

🤖

Let Claude Make Its Own Skills

Say: "I want a skill that does X. Here's how skills work [paste docs]. Build it for me." Test in a fresh instance with zero context. If it fails, give feedback. Usually reaches 95%+ accuracy in 2–3 iterations.

📊

Custom Status Line

In terminal, run /status-line and ask for a token usage loading bar: "Show me a bar with X/200K tokens used." Monitors context burn rate live as you work.

🎯

Thinking Tab = Free Scratch Pad

When extended thinking is on, Claude reasons in a "thinking" tab that doesn't pollute your message context. The output is short. The reasoning is long. You pay for thinking, but your context stays clean.

🌐

Chrome DevTools MCP = Browser Control

Install the Chrome DevTools MCP. Claude can navigate, click, screenshot, fill forms, and extract data from any website — even those with no API. Far faster and more reliable than built-in browser tools.

🔀

Parallel Multi-Tab Development

Open multiple Antigravity windows. Run Claude in each on different design variants or features simultaneously. Cap at 3–4 tabs. Use chime hooks to know when each finishes. You generate 3–4x output in the same time.

⚠ Extension Security: Only install Claude Code extensions from verified Anthropic sources (look for the checkmark). Third-party extensions with similar names have been known to contain malware.

Deployment

Netlify — Static Sites
  • Free tier available
  • Drag-and-drop or Git-connected
  • Best for: websites, web apps, frontends
  • Claude can deploy directly with CLI
  • Add env variables in Site Settings
  • Custom domains supported
Modal — Backend / APIs
  • $5 free credit on signup (often lasts months)
  • Deploy any Python function as a live URL
  • Best for: skills, webhooks, automation endpoints
  • Claude can write and deploy in one step
  • Integrates with N8n, Make.com, Zapier via webhooks
  • Much cheaper than no-code platforms
Security Warning ⚠
  • Never deploy vibe-coded apps with user data without a security review
  • Use non-obvious domain names (scanners find short domains fast)
  • Don't charge money without a dev reviewing auth
  • Use for internal tools and client-facing (not public e-commerce)
  • Have a developer audit for $200–500 before going public

Core Mental Models

Task → Do → Verify

The core loop. Never give Claude a task without also giving it a way to verify its own result — screenshot, test, comparison. This is the single biggest lever for quality.

The Ship Analogy

CLAUDE.md is your initial compass heading. A 1° error at the start becomes a 1,000km miss over long distance. Nail the setup before sailing. Every session benefits.

1 Min Planning = 10 Min Building

The ROI on plan mode is enormous. Every minute spent in the spec phase eliminates many minutes of rebuilding. Always plan before complex builds.

You Are the Ideas, Claude Are the Hands

Your value is knowing what to build and why. Claude's value is building it fast. Don't try to out-code Claude. Out-think it. Give it direction, let it execute.

Carve the River, Float the Boat

Your job is to create constraints (CLAUDE.md, rules, skills, verification loops). Claude flows within those constraints. The tighter the riverbanks, the faster and straighter the water flows.

Token Budget = Quality Budget

More tokens ≠ better results. Longer context actually reduces Claude's average output quality. Ruthlessly managing context isn't just about cost — it's about maintaining peak performance.
